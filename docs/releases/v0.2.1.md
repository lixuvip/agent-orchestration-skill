# v0.2.1 — Adaptive Role Threads

`v0.2.1` makes `agent-orchestration` more adaptive at the moment a role thread is created. The coordinator now chooses a best-fit supported thinking effort for each user-visible thread, while optional `agy` review and research use an explicit consent gate and fail-fast availability cache.

This is a backward-compatible patch release on top of the Lite / Standard / Durable operating model introduced in `v0.2.0`. It does not add another capability pack or recurring process.

## Highlights

### Best-fit thread thinking

- Thinking effort is selected independently from Lite / Standard / Durable routing and from role names.
- The coordinator considers expected quality, ambiguity, reasoning depth, blast radius, verification difficulty, and latency/cost.
- Expected quality, risk coverage, and verification reliability come before efficiency.
- Lower effort is chosen only when adjacent supported levels are equally suitable.
- When the exact effort is unavailable, high-risk or high-ambiguity work rounds up rather than down.
- Explicit user effort wins when supported. The coordinator never changes `model` unless the user requested that model.
- Dispatch, intake, task-board, and examples record requested effort, applied effort, and rationale.
- A creation surface that cannot apply an override records `INHERITED` or `UNSUPPORTED` instead of claiming success.

The router exposes this requirement as:

```text
COORDINATOR_SELECT_BEST_FIT_IF_SUPPORTED
```

### Optional agy consent and fast fallback

- An explicit request for `agy`, Gemini, Antigravity, or an external model authorizes one bounded read-only pass; whole-repository disclosure and writes remain separately authorized.
- When a user asks for an audit or research pass without naming an external model, the coordinator offers `agy` once as an optional auxiliary stream.
- Decline or no confirmation means Codex-only. The coordinator does not probe or invoke `agy` and does not ask again in the same goal.
- After opt-in, `command -v agy` runs once per goal and host.
- A missing binary records `AGY_UNAVAILABLE`, notifies the user once, skips `agy models`, health checks, and `run_agy_print.py`, then continues Codex-only.
- A failed model discovery or health check records `AGY_UNHEALTHY`, notifies once, and does not retry automatically in the same goal/environment.
- Recheck happens only for a new goal, changed host or `PATH`, or an explicit user request after installation or repair.

Capability state lives in coordinator context for Lite, `TASK_BOARD.md` for Standard, and automation memory for Durable.

### Unified runtime and documentation surfaces

- Updated `SKILL.md`, the bilingual coordination runbooks, router metadata, intake and dispatch templates, task-board fields, filled examples, and `agents/openai.yaml`.
- Updated English and Chinese README, quickstart, examples, forward-test scenarios, and changelog.
- Added regression coverage that rejects the former lowest-adequate policy from active runtime and documentation surfaces.
- Kept the entrypoint and capability packs within the existing scale budgets.

## Why This Matters

| Situation | v0.2.1 behavior |
| --- | --- |
| Mechanical inventory or fixed-format extraction | Usually selects `minimal` or `low` when that is the best fit. |
| Normal implementation or bounded debugging | Usually selects `medium`, with an explicit rationale. |
| Architecture, security, cross-repository contracts, or high-risk release review | Can select `high` or `xhigh`; unavailable exact levels preserve risk coverage by rounding up. |
| Ordinary code audit without an external-model request | Offers optional `agy` once and continues Codex-only if not confirmed. |
| Machine without `agy` | Detects once, reports once, and avoids repeated discovery and timeout cost. |

## Upgrade From v0.2.0

No migration is required for existing Lite / Standard / Durable workflows, callback envelopes, automation memory, or project instructions.

If project-local prompts assert the old `COORDINATOR_SELECT_LOWEST_ADEQUATE_IF_SUPPORTED` value or describe thinking selection as "lowest adequate," update them to best-fit semantics. Existing tools that ignore the new router metadata continue to work.

Upgrade and install:

```bash
git pull
./scripts/install.sh
```

The installer validates the clean source, retains the previous copy, writes source provenance, and verifies installed parity. Restore the retained copy with:

```bash
./scripts/install.sh --restore
```

## 中文说明

# v0.2.1 — 自适应角色对话

`v0.2.1` 让 `agent-orchestration` 在创建角色对话时更加自适应。主协调者会为每个用户可见子对话选择最适合且工具支持的思考级别；可选 `agy` 审查与调研则使用明确授权门和快速失败的可用性缓存。

这是基于 `v0.2.0` Lite / Standard / Durable 运行模型的向后兼容补丁版本，不增加新的能力包或常驻进程。

## 主要变化

### 最适合的子对话思考级别

- 思考级别与 Lite / Standard / Durable 路由、角色名称相互独立。
- 协调者综合预期质量、歧义、推理深度、影响范围、验证难度和延迟/成本。
- 预期质量、风险覆盖和验证可靠性优先。
- 只有相邻受支持级别同样适合时，才选择较低档。
- 精确级别不可用时，高风险或高歧义任务向上取档而不是向下取档。
- 用户明确指定且工具支持时优先遵从；未经用户指定，不自行更换 `model`。
- intake、dispatch、task board 和示例都会记录期望级别、实际级别和选择理由。
- 创建位置无法应用覆盖时，记录 `INHERITED` 或 `UNSUPPORTED`，不声称已经成功设置。

路由器返回的新策略值是：

```text
COORDINATOR_SELECT_BEST_FIT_IF_SUPPORTED
```

### 可选 agy 授权与快速降级

- 用户明确要求 `agy`、Gemini、Antigravity 或外部模型时，视为授权一次有边界的只读任务；整仓披露和写操作仍需单独授权。
- 用户只要求代码审计或调研、没有点名外部模型时，协调者只询问一次是否加入 `agy` 辅助流。
- 拒绝或未确认时直接 Codex-only，不探测、不调用，同一目标内不再询问。
- 确认后，每个目标和主机只执行一次 `command -v agy`。
- 未检测到二进制时记录 `AGY_UNAVAILABLE`，只提示一次，跳过 `agy models`、健康检查和 `run_agy_print.py`，直接继续 Codex-only。
- 模型发现或健康检查失败时记录 `AGY_UNHEALTHY`，只提示一次，同一目标/环境内不自动重试。
- 只有新目标、主机或 `PATH` 变化，或用户在安装/修复后明确要求重检时才再次检测。

Lite 把能力状态保存在协调上下文，Standard 写入 `TASK_BOARD.md`，Durable 写入 automation memory。

### 统一运行表面和文档

- 更新 `SKILL.md`、中英文协调手册、路由元数据、intake/dispatch 模板、task board、填充示例和 `agents/openai.yaml`。
- 更新中英文 README、快速开始、示例、前向测试场景和 CHANGELOG。
- 新增回归检查，禁止旧的最低足够策略重新进入运行和文档表面。
- 入口和能力包继续满足原有 scale budget，不因为新增策略而变笨重。

## 实际价值

| 场景 | v0.2.1 行为 |
| --- | --- |
| 机械盘点或固定格式提取 | 当它最适合任务时，通常选择 `minimal` 或 `low`。 |
| 常规实现或有边界调试 | 通常选择 `medium`，并记录明确理由。 |
| 架构、安全、跨仓库契约或高风险发布审查 | 可以选择 `high` 或 `xhigh`；精确档位不可用时优先保持风险覆盖。 |
| 未点名外部模型的普通代码审计 | 只询问一次是否使用 `agy`，未确认就继续 Codex-only。 |
| 本机没有安装 `agy` | 只检测一次、提示一次，避免重复发现和超时成本。 |

## 从 v0.2.0 升级

现有 Lite / Standard / Durable 工作流、回调信封、automation memory 和项目指令不需要迁移。

如果项目内 prompt 写死了旧的 `COORDINATOR_SELECT_LOWEST_ADEQUATE_IF_SUPPORTED`，或者仍描述为“最低足够思考级别”，需要更新成 best-fit 语义。忽略新路由元数据的旧工具仍可继续运行。

升级并安装：

```bash
git pull
./scripts/install.sh
```

安装器会验证干净来源、保留上一版本、记录来源信息并校验安装一致性。如需恢复保留的上一版本：

```bash
./scripts/install.sh --restore
```

验证命令与英文部分相同，包含 static、smoke、forward、protocol、automation、routing、scale、内置 skill 校验和 `git diff --check`。
