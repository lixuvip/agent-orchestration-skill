# Communication Protocol

本文件定义协调者与角色对话之间的沟通规则。

## 基本规则

1. 每个任务只能有一个明确负责人。
2. 每个任务必须声明可编辑范围和只读范围。
3. 每个任务必须声明禁止事项。
4. 每个角色必须按固定格式回复。
5. 协调者不能把未验证的角色输出直接交付给用户。
6. 角色不能把“我认为可以”当作验证结果；必须说明实际运行过什么命令，结果是什么。
7. 异步多对话任务必须有回调或巡检机制，不能依赖角色回复自然出现在协调者上下文里。

## 标准状态

| 状态 | 含义 | 协调者动作 |
| --- | --- | --- |
| `DONE` | 任务完成，并按要求验证。 | 检查输出，流转到下一角色。 |
| `DONE_WITH_CONCERNS` | 任务完成，但存在风险、不确定点或未覆盖场景。 | 先处理风险，再决定是否流转。 |
| `NEEDS_CONTEXT` | 缺少信息，无法继续。 | 补充上下文或缩小任务范围。 |
| `BLOCKED` | 被权限、环境、冲突或需求不清阻塞。 | 解决阻塞或升级给用户。 |

## 必须停止并报告的情况

任何角色遇到以下情况必须停止：

- 需要密钥、Token、私有证书、付费账号、外部登录。
- 需要执行生产部署、数据库变更、强推、重置、清理大量文件。
- 需要下载大模型、大数据集、门控资源或高成本资源。
- 发现工作区有冲突性未提交改动。
- 发现任务目标与项目文档或用户要求冲突。
- 验收标准不清楚，继续做只能靠猜。

## 下发任务时必须包含

- `Role`：本次角色。
- `Goal`：一句话目标。
- `Context`：任务背景和相关约束。
- `Editable scope`：可修改范围。
- `Read-only scope`：只能查看不能改的范围。
- `Out of scope`：明确禁止事项。
- `Verification`：需要执行的验证命令或手工检查项。
- `Delivery format`：要求按哪个模板回复。
- `Stop conditions`：遇到哪些情况必须停。
- `Callback`：协调者对话 ID、是否需要完成后回调、回调格式。
- `Monitoring`：是否由协调者创建周期性巡检自动化。

## 回调与巡检规则

角色完成后，如果当前环境提供对话发送工具，应向协调者对话发送完成回调。回调必须包含状态、摘要、验证和风险。

如果角色无法回调，必须在自己的最终回复中写明：

```text
CALLBACK_FAILED: <REASON>
```

协调者不能只依赖回调。只要存在两个以上异步角色对话，协调者必须记录所有对话 ID，并优先创建 5 分钟巡检自动化。巡检自动化负责读取各角色对话，直到所有角色进入终态，再汇总并关闭自身。

## 协调者检查清单

流转到下一角色前，协调者必须检查：

- 回复状态是否是 `DONE` 或可接受的 `DONE_WITH_CONCERNS`。
- 角色是否遵守职责边界。
- 变更是否只发生在允许范围内。
- 验证命令是否真实运行并给出结果。
- 未运行的验证是否说明了原因。
- 是否存在安全、隐私、成本或生产环境风险。
- 是否需要用户确认。

## 角色之间如何交接

角色之间不要直接假设对方知道上下文。协调者负责把上一角色的有效输出压缩成下一任务的上下文。

交接内容至少包括：

- 上一角色完成了什么。
- 哪些文件或行为被影响。
- 已运行的验证。
- 未解决风险。
- 下一角色需要验证或处理什么。
