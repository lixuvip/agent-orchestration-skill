# Requirement Writing Guide

本文件说明协调者如何把用户需求写成可以交给角色执行的任务。

## 好任务的结构

一个合格任务至少包含：

- 背景：为什么要做。
- 目标：做完后用户能得到什么。
- 范围：允许处理哪些文件、模块、页面、接口、流程。
- 禁止范围：明确不能碰什么。
- 验收标准：怎样判断完成。
- 验证方式：用什么命令、手工步骤或证据验证。
- 停止条件：遇到什么情况必须先报告。

## 目标写法

不好的写法：

```text
优化一下设置页。
```

好的写法：

```text
让设置页支持新增、编辑、删除模型配置，并在保存失败时显示明确错误提示。
```

原则：

- 目标必须描述可观察结果。
- 避免“优化”“完善”“处理一下”这类空泛词。
- 如果确实是探索任务，明确写成“调研并输出方案”，不要伪装成实现任务。

## 范围写法

不好的写法：

```text
你看看相关文件，能改就改。
```

好的写法：

```text
可编辑范围：
- src/settings/*
- tests/settings/*

只读范围：
- src/auth/*
- docs/api.md

禁止范围：
- 不修改数据库 schema。
- 不修改登录流程。
- 不新增外部依赖。
```

原则：

- 可编辑范围越明确，冲突越少。
- 只读范围用于给上下文，但不允许角色改动。
- 禁止范围要写具体，不要只写“不要乱改”。

## 验收标准写法

推荐格式：

```text
当 <用户动作或系统条件> 时，系统应该 <可观察结果>。
```

示例：

```text
- 当用户保存一个缺少名称的配置时，系统应该阻止保存并显示“名称不能为空”。
- 当用户删除最后一个配置时，列表应该显示空状态，而不是保留旧数据。
- 当接口返回 401 时，页面应该提示重新登录，不应该无限重试。
```

验收标准应避免：

- “体验更好”。
- “尽量稳定”。
- “应该没问题”。
- “和之前一样”但不说明之前行为是什么。

## 验证方式写法

每条验证都应说明期望结果。

```text
验证：
- npm test -- settings.test.ts
  期望：全部通过。
- npm run build
  期望：构建成功，无类型错误。
- 手工检查：新增、编辑、删除配置。
  期望：三个操作都有成功提示，刷新后状态保持一致。
```

如果某项验证可能无法运行，应提前说明替代方案：

```text
如果本地缺少数据库，请不要创建生产连接；改为报告缺失依赖和需要的本地启动命令。
```

## 停止条件写法

停止条件用于防止 Agent 乱猜或做危险操作。

通用停止条件：

- 需要密钥、Token、生产账号或付费权限。
- 需要下载大模型、大数据集或高成本资源。
- 需要改数据库 schema、执行迁移或部署生产。
- 发现未提交改动与任务冲突。
- 需要修改禁止范围内的文件。
- 验收标准不清，继续执行只能靠猜。

## 拆分任务的标准

应该拆分的情况：

- 任务同时涉及产品设计、工程实现和 QA。
- 任务会修改多个互不相关模块。
- 任务有明显先后顺序。
- 一个角色无法独立完成验收。

不需要拆分的情况：

- 单文件小修。
- 文案调整。
- 明确的小 bug 修复。
- 只需要一次只读诊断。

## 任务命名

推荐格式：

```text
<角色>/<模块>/<动作>
```

示例：

```text
Product/Settings/Define model configuration UX
Engineer/Settings/Implement model CRUD
QA/Settings/Validate model CRUD
Review/Settings/Review model CRUD diff
```
