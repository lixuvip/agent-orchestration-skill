# 轻量协调

仅在多 owner、跨仓库/worktree，或正式 QA、Review、Release 门禁时加载本文件。

## 选择执行面

| 需求 | 执行面 |
| --- | --- |
| 一个 owner 连续使用同一上下文 | 当前任务 |
| 有界独立探索、测试、盘点或总结 | 内部 subagent；结果交回协调者 |
| 用户需要侧边栏独立管理或直接继续交流 | 新建或现有用户任务线程 |
| 新方向需要继承源任务已完成历史 | fork 线程 |
| 并行写入需要 checkout 隔离 | worktree 线程 |
| 同一任务需要在 Local 与 Worktree 间移动 | handoff |

“Agent”“子进程”“子对话”等词只是线索，不是固定工具名。派发前说明是否生成侧边栏新任务、结果回到哪里、谁负责后续交流。

## 有限任务流程

1. 明确结果、边界、owner、写入范围、证据、返回面和停止条件。
2. 预检所需的读取、写入、执行、网络、浏览器和 connector 能力。方案确实存在较大歧义时，先在当前任务中保持只读规划，方向确认后再实现。父任务处于 Plan/只读状态，不代表子 Agent 自动只读。
3. 只拆独立工作；重叠写入必须隔离或串行。默认保持扁平委派树，只有明确有价值且 owner 仍可审计时才嵌套委派。
4. 使用原生 subagent 或 thread 工具派发，只提供完成任务所需的最小输入和能力。
5. 使用原生 Agent/线程状态等待。有限任务不创建回调信封、任务板、heartbeat、租约或轮询文件。
6. 检查结果、产物、diff、真实命令和遗留风险。
7. 退回修正或验收。角色完成只是待检查证据，不自动等于交付。

## 派发与证据契约

简洁的派发应说明：

- 目标和当前产物/baseline；
- 最少必要输入与能力；
- 可编辑、只读和明确不在范围内的边界；
- 预期证据以及结果返回位置；
- 后续 owner 和停止/升级条件。

证据随任务类型变化：调研返回来源、观察到的事实和明确标注的推断；实现返回 diff 和定向测试；Review 返回发现、位置、影响和证据；QA 返回候选产物、场景、预期、实际结果，必要时附原始输出位置。

缺少必要能力时，派发前先调整执行面或范围。不要等任务结束后才发现 owner 无法检查真实产物或运行必需门禁。

## 中途转向与过期结果

活跃工作期间收到用户新消息时，将其分类为：

- **替换**：中断或改派已经不符合新目标的工作；
- **追加**：保留当前目标，只扩展受影响 owner；
- **状态/问题**：回答问题，不暗中改变范围。

当前任务保存最新有效范围。来自旧范围的迟到结果属于过期结果：可以作为背景参考，但未经重新验证，不得合并、验收或当成当前结论交付。

修正通常回到原 owner，并附失败证据和需要变化的 delta。独立 Reviewer 使用新上下文，只接收候选产物、需求和原始证据；不要把实现者的结论预先当成事实灌入。

## 验证预算

- 实现阶段先跑定向测试和低成本静态检查。
- 形成候选产物后再做独立审查。
- 只有风险或用户意图需要时才启用一个独立 Reviewer；默认不嵌套 Reviewer。
- 最终候选只跑一次完整相关套件。后续代码变化只重跑受影响检查和项目强制的最终门禁。
- 不在同一产物上重复执行完全相同的昂贵套件；优先检查原始输出或做更小的独立验证。
- 验收前从完整用户需求、后续补充和修正中生成闭环清单。每项映射到实际动作、当前证据，以及 `done`、`waived` 或 `blocked`；测试通过不能替代未覆盖的需求。

重试必须有上限。每次重试都要新增证据、缩小失败范围或改变方法。同一阻塞在没有新信息的情况下重复出现时，应停止循环并询问、升级或如实报告，不无限消耗。

## 可选 Best Of N

只有用户明确要求多方案，或高歧义确实值得额外成本时，才使用 2-3 个隔离候选。结果返回前先固定评价标准；写入候选全部隔离；用同一证据比较；允许所有候选都不合格。集成胜出候选后，只对它运行一次最终完整套件。

## 正式门禁

合并、推送、部署、发布、安全敏感修改或明确 QA/Review 时：

- 标识精确候选 commit 或非 Git 产物；
- 要求该产物的当前测试/审查证据；
- 后续修改会使受影响证据失效；
- 实现权限与 merge/push/deploy 权限保持分离；
- 如实报告阻塞和豁免。

交接使用简洁自然语言，包含 owner、范围、产物、验证、风险和下一步；除非外部系统明确要求，不生成机器事件信封。

在 fork、handoff、长时间暂停或强依赖上下文的继续工作前，留下恢复胶囊：目标、最新用户约束、产物 baseline、已验证证据、已做决策、待办、活跃后台工作、阻塞和精确下一步。使用原生任务历史保存；只有项目确实需要长期产物时才写入仓库文件。

最终交付前，审计原生状态中的活跃子 Agent、用户自有任务、后台命令、monitor 和 automation。逐项等待、取消或明确报告。仍有未盘点工作可能改变结果时，不得暗示整体目标已经完成。

## 沟通

原生活动 UI 是主要进度面。只有出现有效状态变化、阻塞、授权请求、风险变化或新证据时才补充 commentary；不要反复播报“仍在运行”和未变化的安全边界。

原生能力不可用时，留在当前任务或使用简短人工交接，并说明限制，不伪装独立工作正在运行。
